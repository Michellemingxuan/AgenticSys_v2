OpenAIAdapter
Dev · direct API
Bypass path
Thin wrapper over openai.chat.completions.create. No firewall simulation. Uses the platform's native function-calling — tools become tools=[{type:"function",...}]. Used locally for iteration.
Sends [system, user] messages as-is.
Native tool calls: OpenAI returns tool_calls → adapter dispatches to Python functions → role:"tool" response appended.
No digit masking, no keyword filtering, no role-label rewriting.
Does not raise FirewallRejection — FirewallStack effectively passes through.
SafeChainAdapter
Deployment · firewall-wrapped
Hardened path
Runs inside the SafeChain LLM pipeline (ValidChatPromptTemplate). All firewall avoidance happens here — every message is transformed before the LLM sees it.
Collapses all messages into a single human message (SafeChain requirement).
Uses neutral role labels: Context: / Request: / Response: — no [SYSTEM] markers.
Pre-masks 8+ digits, filters exec/eval/import/__dunder__.
No native function-calling — tools described in text; model responds with JSON the adapter parses in a loop.
401 → token refresh + local retry · 403/400 → FirewallRejection for upstream retry.

Seven avoidance techniques What · when · why it helps
Technique When Why it avoids the firewall
Neutral role labels
Context: / Request: / Response:		 pre-invoke SafeChain pattern-matches [SYSTEM] / [USER] / [ASSISTANT] as role-injection markers. Neutral nouns don't trigger.
Collapse to single human message
all roles merged into one string		 pre-invoke SafeChain's ValidChatPromptTemplate accepts only ("human", "{__input__}"). Also removes role-juncture patterns entirely.
Mask 8+ digit runs
\d{8,} → ***MASKED***		 pre-invoke Account numbers / long IDs trip PII detectors. Masking at send-time keeps the LLM from ever seeing the raw value.
Filter code-exec keywords
exec / eval / import / __dunder__ → [FILTERED]		 pre-invoke Prevents RCE-lookalike signatures in tool results from triggering SafeChain's code-execution heuristics.
Prompt-injected tool calling
tools described in text · model emits JSON		 tool loop SafeChain's wire format is pure text in / text out. Native function-calling would bypass the sanitizers entirely — not allowed.
401 silent refresh
detect "401" in error → _refresh_llm() + retry		 on error Token expiry during long sessions doesn't surface as a firewall block — adapter refreshes transparently.
403/400 → FirewallRejection
translate HTTP → typed exception		 on error Gives the upstream FirewallStack a uniform signal to retry with guidance + extra sanitization.
Tool calling · native vs prompt-injected The biggest behavioral difference between the two adapters
OpenAIAdapter · native
Uses platform function-calling
Tools become structured JSON Schema sent alongside the prompt. The API returns tool_calls as a typed field; the adapter dispatches, appends role:"tool", and loops.
# tool definition (sent as structured field) tools=[{ "type": "function", "function": { "name": "query_table", "parameters": { /* JSON Schema */ } } }] # response (typed) choice.message.tool_calls[0].function.name choice.message.tool_calls[0].function.arguments
SafeChainAdapter · prompt-injected
Tool schema embedded in system prompt · JSON response parsed
Tool signatures are formatted as text and appended to the system prompt. The LLM is instructed to respond with only a specific JSON shape. The adapter parses each response, dispatches to the Python function, and injects the result back as a user message.
# appended to system prompt (plain text) You have access to the following tools. To call a tool, respond with ONLY this JSON: {"tool_call": {"name": "...", "arguments": {...}}} When final: {"output": {...}} Available tools: - query_table(table_name: string, ...) Query a data table for the current case.
Execution lifecycle · where the code runs · LLM status at each beat All tool code runs in Python · the LLM is stateless between invocations
Code runs in Python
The LLM never executes code. It emits a JSON {"tool_call":...}; the adapter dispatches to a regular Python function (query_table, get_table_schema, …) and captures the return value.
LLM has no memory
Every _invoke() call is independent. Context lives only in the messages list maintained by the adapter; each round feeds the full history back.
Status is binary
The LLM is either inferring during _invoke(), or idle. No background processing, no warm state.
