Agentic Case Review System — Defense In Depth

Compliance Is Not a Single Gate

Compliance is enforced at every layer of the system, creating defense in depth. All rules are driven by the pillar YAML — the same file that controls specialist focus, table scope, and prompt overlays also declares blocked_fields. No code changes required to add, remove, or adjust compliance rules. Edit the YAML, and every layer enforces the new rule automatically.

Defense In Depth — Four Enforcement Points
Layer 1 — Orchestrator
Pre-Routing Filter
Before any specialist is called. The Orchestrator reads blocked_fields from the pillar YAML and checks the reviewer's question. If the question requests blocked data, the Orchestrator rewrites or rejects the request before routing to any specialist. This is the first line of defense — no blocked data ever enters the specialist pipeline.
YAML source
blocked_fields: ["bureau_raw_pii", "salary_data"]
↓
Layer 2 — Specialists
Data Request Construction
When building a Data Request. Each specialist carries the blocked_fields and table_scope context from the pillar YAML (injected at call time by the Orchestrator). The specialist will not include prohibited fields in any Data Request it constructs — it self-censors based on the YAML context in its prompt.
Injected at call time
blocked_fields: ["bureau_raw_pii", "salary_data"]
table_scope: ["bureau_full"]
↓
Layer 3 — Repository Agent
Hard Enforcement + Audit
Every request, no exceptions. The Repository Agent validates every field in every Data Request against the pillar YAML blocked_fields. It writes an audit log entry for every request — approved or rejected. It also enforces table_scope to ensure the specialist only accesses tables it's authorized for under this pillar. Finally, it redacts PII and sensitive fields from the data before forwarding to Data Agents.
Enforced fields
blocked_fields: ✕ hard reject
table_scope: ✕ scope gate
audit_log: ✓ every request
↓
Layer 3 — Data Agents
Downstream Only
No direct access to raw data. Data Agents only ever receive pre-validated, redacted requests from the Repository Agent. They have no knowledge of the pillar YAML, no access to raw schemas, and no visibility into which specialist made the request. They are stateless execution engines — compliance is not their concern because it has already been enforced upstream.
Visibility
No YAML access.
No raw schemas.
Redacted data only.

Adding a new blocked field = add one line to the pillar YAML. Every layer (Orchestrator, Specialists, Repository Agent) enforces it automatically. No code changes, no redeployment — just a config change.

Bureau Specialist — Data Request Construction
Specialist carries blocked_fields: ["bureau_raw_pii", "salary_data"] in its prompt context. Constructs a Data Request for bureau_full table — does not include salary_data in the requested columns because it's in its blocked list.

Repository Agent — Hard Enforcement
Even if the specialist somehow included a blocked field (defense in depth), the Repository Agent cross-references every requested field against blocked_fields. In this case, the request is clean — but the audit log records the request regardless.
Validated ✓ Audit Logged ✓

Data Agent — Executes Clean Request
Receives a redacted, validated request. Runs Code Gen → Execute → Retrieve. Returns structured results back through the Repository Agent to the Bureau specialist. No salary data ever touched.
Clean Data Returned ✓
Key principle: Compliance is not a checkbox at one gate — it is a YAML-driven thread that runs through every layer of the system. The Orchestrator filters before routing, Specialists self-censor during Data Request construction, the Repository Agent hard-enforces and audits, and Data Agents only ever see clean data. All rules come from one source: the pillar YAML file.
