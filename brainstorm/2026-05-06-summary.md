*Combines: tool-calls-and-redaction, team-construction, data-query (2026-05-06)*

# Part I: Tool Calls, Redaction, Firewall & Skill Loading

---

## 1. The Core Problem

LLMs can only generate text. They cannot browse the web, query a
database, or check live data. **Tool calls** are the bridge between
"text generation" and "real-world actions." The LLM never runs code
itself — it *requests* that your code runs something on its behalf.

---

## 2. The Mental Model: A Request–Execute–Report Loop

\newpage

```
Client code
|
+- defines tool schemas  ----------------------------------------+
+- sends [messages + tools] to API  -->  LLM reasons; picks tool |
|                                        |                        |
|  <-- tool_call response ---------------+                        |
|                                                                 |
+- executes real function(s)                                      |
+- appends tool result to messages                                |
+- sends back to API  -->  LLM sees result -> writes final answer |
                                                                  |
         ^                                                        |
         +-- loop repeats if LLM needs more tools ----------------+
```

\newpage

Think of it as an assistant who can ask you to look things up:

```
You  -> LLM: "What's the weather in NYC?"
LLM  -> You: "Please call get_weather('NYC') and give me the result."
You  -> (executes get_weather) -> "72 degrees F, sunny"
You  -> LLM: "Tool result: 72 degrees F, sunny"
LLM  -> You: "It's 72 degrees F and sunny in New York City!"
```

---

## 3. The Mechanism — Five Steps

### Step 1 — Define tools (JSON schema, not code)

```python
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string"}
            },
            "required": ["city"]
        }
    }
}]
```

This is a *description* — the LLM reads it to know what capabilities
exist. No code runs here.

### Step 2 — LLM responds with a `tool_calls` block (not text)

```json
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_abc123",
    "type": "function",
    "function": {
      "name": "get_weather",
      "arguments": "{\"city\": \"NYC\"}"
    }
  }]
}
```

Instead of answering directly, the model emits a structured request.
Arguments are a JSON string — typed, validated, safe to parse.

### Step 3 — Your code executes the real function

```python
if response.choices[0].message.tool_calls:
    for tc in response.choices[0].message.tool_calls:
        result = get_weather(json.loads(tc.function.arguments)["city"])
```

### Step 4 — Return the result as a `tool` role message

```python
messages.append({
    "role": "tool",
    "tool_call_id": "call_abc123",
    "content": "72 degrees F, sunny"
})
```

### Step 5 — Call the API again; LLM produces the final answer

The LLM now has the data it needed and writes a normal text response.

---

## 4. Key Design Choices

\newpage

| Design choice | Reason |
|---|---|
| LLM outputs a schema request, not executable code | Sandboxing — the model never runs arbitrary code |
| Tool result sent as a new conversation turn | Keeps full history; model sees all context |
| `tool_call_id` links request to result | One response can request **multiple tools in parallel** |
| Arguments are a JSON string | Typed, validated input; prevents injection |

\newpage

## 5. The Redaction Step

In production (SafeChain path), **every message is sanitized before it
leaves the client** — before hitting the firewall, before the LLM sees
it.

### What it does

Located in `llm/firewall_stack.py -> sanitize_message()` and applied
by `llm/safechain_client.py -> _redact_message()`:

```python
_CASE_ID_RE = re.compile(r"CASE-\d+")
_DIGIT_RUN_RE = re.compile(r"\d{6,}")

def sanitize_message(message: str) -> str:
    masked = _CASE_ID_RE.sub("[CASE-ID]", message)
    return _DIGIT_RUN_RE.sub("***MASKED***", masked)
```

| Pattern | Masked as | Example |
|---|---|---|
| `CASE-123456` | `[CASE-ID]` | Case identifiers |
| Any 6+ digit run | `***MASKED***` | Account numbers, SSNs, phone numbers |

### Where it sits in the pipeline

\newpage

```
Agent / SDK
    |
    |  messages = [system, user, tool_result, ...]
    |
    v
_redact_message()          <- sanitize every outbound message
    |
    |  long digits -> ***MASKED***
    |  CASE-\d+   -> [CASE-ID]
    v
_combine_messages()        <- flatten multi-turn -> single string
    |                         (SafeChain: only one human message)
    |
    v
SafeChain Firewall         <- HTTP POST; firewall evaluates content
    |
    v
LLM                        <- sees only masked identifiers
```

\newpage

### Why before flattening?

1. **No PII leaks through concatenation** — joining messages before
   masking could create new 6-digit sequences at join boundaries.
2. **Retry safety** — `_inject_guidance()` also re-redacts every message
   when it appends firewall guidance on a retry, so redaction is
   idempotent across attempts.

### The `redact_payload` helper (deep recursion)

For tool *results* (Pydantic models, dicts, lists),
`firewall_stack.redact_payload()` walks the entire structure:

```python
def redact_payload(payload):
    if isinstance(payload, str):    return sanitize_message(payload)
    if isinstance(payload, dict):   return {k: redact_payload(v) ...}
    if isinstance(payload, list):   return [redact_payload(v) ...]
    if isinstance(payload, BaseModel):
        return Type.model_validate(redact_payload(payload.model_dump()))
    return payload
```

Even nested tool outputs — e.g., a `SpendingReport` with embedded
account numbers — are fully masked before the LLM reads them.

## 6. Firewall Handling

After redaction, every API call passes through a firewall layer.

| Implementation | File | Used when |
|---|---|---|
| `FirewalledAsyncOpenAI` | `llm/firewall_client.py` | Dev / direct OpenAI path |
| `SafeChainChatCompletions` | `llm/safechain_client.py` | Prod / SafeChain path |

`FirewallStack` owns the shared state: logger, `max_retries=2`, and
`asyncio.Semaphore(concurrency_cap=3)`.

### The retry-with-guidance loop

```python
messages = [_redact_message(m) for m in messages]   # 1. pre-redact
attempt = 0
while True:
    try:
        async with firewall.semaphore:               # 2. acquire slot
            return await base.create(messages=messages, ...)
    except FirewallRejection as e:
        if attempt >= firewall.max_retries:
            raise                                    # 3a. give up
        attempt += 1
        messages = _inject_guidance(messages)        # 3b. inject hint, retry
```

`_inject_guidance` re-redacts all messages and appends
`FIREWALL_GUIDANCE` to the system message so the LLM knows why it was
blocked.

### HTTP error mapping (SafeChain path)

| HTTP status | Meaning | Action |
|---|---|---|
| 401 | Token expiry | Refresh model; retry once immediately (outside the loop) |
| 403 | Firewall block | `raise FirewallRejection("403", ...)` |
| 400 | Bad request | `raise FirewallRejection("400", ...)` |

### Firewall flow

\newpage

```
messages (redacted)
      |
      | async with semaphore      <- cap concurrent calls
      v
  API call
      |
      +-- success --> return ChatCompletion
      |
      +-- FirewallRejection
            |
            +-- attempts < max_retries?
            |         |
            |         yes --> _inject_guidance() --> retry
            |
            +-- no --> log firewall_blocked --> raise
```

\newpage

## 7. Combined Picture (SafeChain prod path)

\newpage

```
 1. User question arrives
 2. Orchestrator composes messages + tool schemas
 3. -- REDACTION -- every message sanitized (digits, case IDs masked)
 4. Messages flattened into one string (SafeChain: one human message)
 5. SafeChain firewall evaluates; blocks or passes
 6. LLM emits {"tool_call": {...}} JSON  (no native function-calling)
 7. SafeChain shim parses JSON -> synthesizes OpenAI tool_calls object
 8. Agent SDK dispatches real tool (data query, etc.)
 9. Tool result -> -- REDACTION -- redact_payload() on the result struct
10. Result appended as "Tool result:" label in next flattened message
11. Steps 4-10 repeat until LLM emits {"output": {...}}
12. Final answer returned to user
```

\newpage

## 8. How Markdown Skills Are Loaded

Every skill is a `.md` file with a YAML frontmatter block and a body.
The frontmatter declares metadata; the body becomes part of an agent's
system prompt or a tool description.

**Frontmatter fields:**

| Field | Values | Meaning |
|---|---|---|
| `name` | string | Human-readable identifier |
| `description` | string | Used as the tool description when mode=tool |
| `type` | `workflow`, `domain`, `helper` | Folder the file lives under |
| `owner` | list of agent names | Which agents claim this skill |
| `mode` | `inline`, `tool` | How the body reaches the LLM |

`load_skill(path)` parses the frontmatter fence, `yaml.safe_load`s it,
separates reserved keys from `meta`, and returns a typed `Skill` object.
`load_skills_for(agent_name)` `rglob`s the tree and filters by `owner`.

### Two delivery modes

**`mode: inline`** — body concatenated into the system prompt via
`render_inline_prompt()`. The LLM reads it as standing instructions on
every turn.

**`mode: tool`** — body set as `wrapper.__doc__` in
`agent_factories/helper_tools.py` via `_with_doc(fn, skill_body)`.
`bind_tools()` reads `__doc__` and uses it as the tool description in
the API request.

### End-to-end skill loading picture

\newpage

```
skills/workflow/team_construction.md
      |
      | load_skill()          <- parse frontmatter + body
      v
Skill.body  (plain string)
      |
      | _compose_orchestrator_instructions()
      v
"=== Team Construction ===\n<body>\n\n---\n\n=== Data Catalog ===\n..."
      |
      | Agent(instructions=...)
      v
System prompt  <-- LLM reads this on every turn


For tool-mode skills:

Skill.body
      |
      | _with_doc(acropedia_lookup, body)
      v
wrapper.__doc__ = body
      |
      | model.bind_tools([wrapper, ...])
      v
Tool description in the API request  <-- LLM reads this when deciding
                                          whether to call the tool
```

\newpage

### Redaction layer breakdown

| Layer | What it does | When it runs |
|---|---|---|
| `redact.md` (inline skill) | Instructs the LLM to mask in its own output | At generation time |
| `sanitize_message` (Python) | Masks inbound text before it reaches the LLM | Before every API call |
| `redact_payload` (Python) | Masks structured tool results recursively | After every tool execution |

## 9. When Does the Specialist See the Tool Description?

The specialist is built with `data_query.md` as its system prompt and
the `@function_tool` callables as its `tools=` list. Both reach the LLM
in **every API request, simultaneously**.

### Request structure

\newpage

```
API request
+-- messages[0]  (system)
|     +-- _compose_instructions()
|           +-- data_query.md body        <- tool guidance prose
|           +-- Domain: <skill.name>
|           +-- Data hints, risk signals, ...
|
+-- tools: [
      { name: "query_table",
        description: "<wrapper __doc__>", <- @function_tool schema
        parameters: { ... }              },
      { name: "aggregate_column", ... },
      ...
    ]
```

\newpage

### Strategic vs. tactical split

| Source | Layer | Purpose |
|---|---|---|
| `data_query.md` body in system prompt | Strategic | Routing rules: when to use each tool, ordering |
| `@function_tool` wrapper `__doc__` in tools array | Tactical | Argument shapes, parameter constraints |
| `_xxx_impl` docstring | Developer only | Internal logic, never sent to the API |

Both the system prompt and the tools array are billed as input tokens on
every turn — before a word of user question is counted.

# Part II: Team Construction — Spending Pattern Walkthrough

This section walks the **team-construction** decision: how the
orchestrator picks which specialists to call, frames each one's
sub-question, and enforces the multi-round protocol that ends in a
`FinalAnswer`. It uses the spending-pattern question as the running
example.

Team construction is **not** a separate stage with its own JSON output.
It IS the orchestrator's first turn — the decision is *expressed* by
which tool calls the orchestrator emits, in what shape, and in parallel.

---

## 11. Where Team Construction Happens

\newpage

```
                    +-------------------------------------------+
 user question -->  |  Orchestrator Agent                       |
 "spending pattern  |    instructions = composed prompt         |
  for customer?"    |    tools = [spec1, spec2, ..., specN,     |
                    |             report_agent, general_spec]   |
                    |    tool_choice = "required"  (1st turn)   |
                    +---------------------+---------------------+
                                          |
                                          | TEAM-CONSTRUCTION DECISION
                                          | (round 1 -- parallel calls)
                                          v
            +---------------+--------------+---------------+
            |               |              |               |
            v               v              v               v
    spend_payments      modeling      report_agent    (no others --
    (sub_question)   (sub_question)  (sub_question)    routing rule
                                                       rejected them)
            |               |              |
            +---------------+--------------+
                            |
                            | ROUND 2 -- mandatory
                            v
                   general_specialist
                   (compares spend_payments
                    + modeling outputs)
                            |
                            v
                      FinalAnswer
```

\newpage

Two passes through the orchestrator LLM:

1. **Round 1**: emit a parallel tool-call batch — *N domain specialists
   + `report_agent`*. The team is built here.
2. **Round 2**: emit `general_specialist(...)` (mandatory whenever
   round 1 used 2+ domain specialists).

Only after round 2 is the orchestrator allowed to emit a `FinalAnswer`.

---

## 12. What the Orchestrator Sees at Decision Time

`_compose_orchestrator_instructions` joins five blocks:

1. `skills/workflow/team_construction.md` — routing rules
2. `skills/workflow/data_catalog.md` — table-level grounding
3. `skills/workflow/synthesis.md` — downstream merge contract
4. `skills/workflow/balancing.md` — report vs. team reconciliation
5. **TOOL-USE DISCIPLINE block** — the hard protocol gates

Then optionally: pillar `concept_glossary` and the auto-generated
**TEAM ROSTER** (`_render_team_roster`), which for each specialist
surfaces its `description`, `data_hints` (table ownership), and top 3
`risk_signals`. Without this roster the orchestrator would route blind.

A second enrichment hits each specialist's tool-description string via
`_describe_specialist(...)`:

```
Domain specialist 'spend_payments'. Spend & Payments -- payment trends,
delinquency, spend spikes. Owns data tables: txn_monthly, spends,
payments. Call with a focused sub-question scoped to this domain.
```

### The TOOL-USE DISCIPLINE block — hard gates

Three rules from `orchestrator_agent.py:73-108`:

1. **Grounding gate** — every `FinalAnswer` MUST be preceded by (a) at
   least one `report_agent` call AND (b) at least one domain specialist.
2. **Parallel execution** — `report_agent` and every selected domain
   specialist must be emitted in a SINGLE response.
3. **General-specialist gate** — when 2+ domain specialists are on the
   team, `general_specialist` MUST be called in round 2 after their
   results land.

### What's in the `tools` array

```python
tools = [redacting_tool(s, name=s.name, description=_describe_specialist(s))
         for s in specialists]
tools.append(redacting_tool(report_agent, ...))
tools.append(redacting_tool(general_specialist, ...))
```

`tool_choice="required"` forces the first orchestrator turn to be a
tool call. The SDK default `reset_tool_choice=True` flips this back to
`"auto"` after the first call so synthesis is unblocked.

---

## 13. The Routing Decision for "Spending Pattern"

The cross-domain row in `team_construction.md:31` is the deciding rule:

> **spending / spend pattern / spend behavior / spend trajectory / spend
> volume / merchant concentration**
>
> MUST include BOTH `spend_payments` AND `modeling`
> (+ `crossbu` only when the question is explicitly B2B).

Three consequences:

- **Team size = 2 specialists.** The cross-domain table overrides the
  "1-2 is normal" default — `spend_payments` alone is incomplete.
- **`crossbu` is excluded.** The question doesn't mention B2B/charge
  volume; the edge-case block reinforces this (balance != spend).
- **Other specialists dropped.** Selection rule #2 forbids "for context"
  picks.

### Sub-question framing per specialist

\newpage

| Specialist | Framing principle | Example sub-question |
|---|---|---|
| `spend_payments` | Use `spends_data` and `payments` vocabulary; ask the broad pattern question because the domain skill's playbook already covers temporal shape + merchant concentration + spend-vs-payment ratio. | *"Walk through the customer's spending pattern: monthly volume + count, top recurring and high-value merchants by name, industry mix, per-merchant trends, and spend-vs-successful-payments ratio."* |
| `modeling` | Use `model_scores` and `score_drivers` vocabulary; ask what the ML scores see. | *"What do the internal model scores say about this customer's spending pattern? Cover spend-related drivers, score evolution over the spend window, divergence between model and bureau."* |
| `report_agent` | Curated-report lookup; near-verbatim of the root. | *"Pull any prior curated narrative about the customer's spending pattern, recurring merchants, or unusual spend behavior."* |

\newpage

The framings are non-overlapping: `spend_payments` works the transaction
tape; `modeling` works the ML signal; `report_agent` works the prior
narrative. Selection rule #4 ("orthogonal across specialists") is the
formal source of this discipline.

---

## 14. Round 2 — `general_specialist`

After all three round-1 results land, the hard gate fires. Because the
team had 2+ domain specialists, the orchestrator MUST emit:

```
general_specialist(sub_question="<spend_payments findings>;
                                 <modeling findings>;
                                 compare and surface contradictions")
```

`general_specialist` is tool-less (`tools=[]`) — pure synthesis on
in-context outputs. It focuses on: does `modeling`'s out-of-pattern
spend signal align with the spend spike `spend_payments` flagged? If
contradictions exist, they must appear in the orchestrator's `flags`.

---

## 15. Compliance Self-Check Before `FinalAnswer`

```
count unique domain specialists called this turn  ->  N
if N >= 2:
    scan conversation for a general_specialist tool result
    if absent: call general_specialist FIRST, then FinalAnswer
    if present: proceed to FinalAnswer
if N == 1:
    skip general_specialist
    proceed to FinalAnswer
```

---

## 16. What Can Go Wrong

\newpage

| Failure | Caught by |
|---|---|
| Routing only `crossbu` for a spending-pattern question | `team_construction.md:39` edge-case block + cross-domain table forcing `spend_payments` |
| Answering with `spend_payments` alone (omitting `modeling`) | Cross-domain row explicitly calls this incomplete; selection rule #1 says read this table BEFORE applying defaults |
| Emitting `FinalAnswer` after only one specialist (no `report_agent`) | TOOL-USE DISCIPLINE rule #1 — both required |
| Skipping `general_specialist` after 2 domain specialists | Compliance self-check at `orchestrator_agent.py:99-107` |
| Adding `bureau` "for context" | Selection rule #2 — every pick must carry weight |
| Reshuffling the team for a follow-up | Selection rule #4 — reuse prior team so per-specialist memory is preserved |
| Hallucinating a `FinalAnswer` from prompt text alone | `tool_choice="required"` forces a tool call on the first turn |

## 18. Build-Time Wiring

The graph is constructed once per session by `Orchestrator.__init__`:

1. **Domain skills discovered** — `list_domain_skills()` globs
   `skills/domain/*.md`. Relevant ones: `spend_payments.md`,
   `modeling.md`.
2. **Specialists built** — `build_specialist_agent(skill, pillar, model)`
   creates an `agents.Agent` whose `instructions` concatenate:
   `data_query.md` body + domain skill body + hints/signals + pillar
   config. `tool_choice="required"` forces the first turn to be a tool
   call.
3. **Report and general specialists built** — curated-md reader and
   cross-domain comparator.
4. **Orchestrator assembled** — each specialist is wrapped as a tool via
   `redacting_tool(...)`, then `build_orchestrator_agent` composes the
   full instructions from four workflow skills + the protocol block +
   the auto-generated roster.

`tools/data_tools.py` is bound via `init_tools(gateway, catalog,
logger)` before the orchestrator runs. The `LocalDataGateway` walks
`data_tables/<case_id>/*.csv` and merges `payments_success.csv` +
`payments_returns.csv` into a single `payments` table with a synthetic
`payment_status` column.

## 20. Inside the `spend_payments` Specialist

The specialist's `instructions` are composed from:

1. **`data_query.md`** — base playbook: six tools, routing rules, the
   windowed-answer template, anti-hallucination rules.
2. **`spend_payments.md`** — domain playbook: four dimensions a pattern
   answer must cover, exact tool call for each.

### Spending-pattern tool call plan

\newpage

| Section | Dimension | Tool call |
|---|---|---|
| A.1 | Volume per month | `summarize_trend('spends', 'Amount', 'Date', period='month', op='sum')` |
| A.2 | Txn count per month | `summarize_trend('spends', 'Amount', 'Date', period='month', op='count')` |
| B.4 | Recurring merchants | `summarize_by_group('spends', 'Amount', 'Merchant Name', op='count', top_n=5, sort_by='count')` |
| B.5 | High-value merchants | `summarize_by_group('spends', 'Amount', 'Merchant Name', op='sum', top_n=5)` |
| B.6 | Per-merchant trend | `summarize_trend('spends', 'Amount', 'Date', period='month', op='sum', filter_column='Merchant Name', filter_value='<name>')` x 3-5 |
| B.7 | Industry mix | `summarize_by_group('spends', 'Amount', 'Merchant Industry', op='sum', top_n=10)` |

## 23. Cross-Cutting Concerns

- **Boundary redaction.** Three layers: `sanitize_message` (input to
  each specialist call), `redact_payload` (output of each specialist
  call), `_format_aggregate` (thousand separators on all numeric results
  so values survive the `\d{6,}` mask).
- **Canonical name resolution.** `_resolve_real_table` and
  `_resolve_real_column` cascade through exact match, catalog aliases,
  `<name>_data` convention, and normalized fuzzy — so specialists use
  canonical names from skill `data_hints` while real CSVs may differ.
- **Per-specialist memory.** `AppContext._specialist_histories` lets
  follow-up questions continue the same specialist thread instead of
  restarting from cold.
- **Dedup cache.** `AppContext._specialist_call_cache` prevents
  identical sub-questions from firing twice in the same context.
- **Forced first turn.** `ModelSettings(tool_choice="required")` on
  both orchestrator and specialists; `reset_tool_choice=True` (SDK
  default) flips back to `"auto"` after the first tool call.
- **Pillar cut-off date.** Injected into every specialist prompt;
  all time-window language is anchored to `cut_off_date`, never today.
