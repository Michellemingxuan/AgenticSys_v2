Prompt Adjustment Guide

Agentic Case Review · Where to edit what
How to change what the agents say to the LLM. Prompts live in three layers: a single base YAML, per-pillar YAMLs, and per-specialist Python skills — plus a handful of hardcoded constants for orchestration agents.

Section 2

Assembly order

Two assembly points, both in config/report_loader.py

Specialist prompt

Called by: get_specialist_prompt(mode, question, findings, domain, pillar_report_instructions)
Injected at: Step-3 user message in agents/base_agent.py
Composition order
  1. prompts.yaml / common.format_instructions
  2. prompts.yaml / {chat|report}.specialist  ← $question, $findings, $domain substituted
  3. pillar.yaml / specialists.{domain}.report_instructions
Joined with \n\n.

Synthesis prompt

Called by: get_synthesis_prompt(mode, pillar_report_format, pillar_synthesis_report)
Injected at: system prompt for Orchestrator.synthesize() in orchestrator/orchestrator.py
Composition order
  1. prompts.yaml / {chat|report}.synthesis
  2. pillar.yaml / report_format  ← top-level, optional
  3. pillar.yaml / synthesis_report.instructions
  4. pillar.yaml / synthesis_report.{section_key}.report_instructions  ← per section
  5. pillar.yaml / synthesis_report.formatter_report
Joined with \n\n.
Section 3

Common edit scenarios

You want to… Edit this file
Make chat answers use dash-bullets instead of paragraphs config/prompts/prompts.yaml → chat.synthesis
Make all specialists cite a cut-off date differently config/prompts/prompts.yaml → common.format_instructions
Change what the bureau specialist reports for credit_risk config/pillars/credit_risk.yaml → specialists.bureau.report_instructions
Change the bureau specialist's expertise framing (every pillar) skills/domain/bureau.py → system_prompt and risk_signals
Add a new specialist (a) new skills/domain/new_domain.py returning DomainSkill(...)
(b) register in skills/domain/loader.py → _DOMAIN_MODULES
(c) add per-pillar overlay in config/pillars/*.yaml under specialists.new_domain
Change a specialist's data tables skills/domain/<name>.py → data_hints
Change how the orchestrator picks specialists / splits into sub-questions orchestrator/orchestrator.py → PLAN_TEAM_PROMPT constant
Change the synthesis merge behavior for every pillar config/prompts/prompts.yaml → chat.synthesis or report.synthesis
Change firewall-retry guidance (what the model sees after a block) gateway/firewall_stack.py → FIREWALL_GUIDANCE
Change the chat-follow-up system prompt orchestrator/chat_agent.py → CHAT_SYSTEM_PROMPT

Hardcoded prompts

Python string constants
Constant File Role

Section 5

Testing a prompt change

# 1. Edit the relevant file above.

# 2. If running in a notebook kernel, reload the YAML cache:
from config import report_loader
report_loader.reload()
#    (.py constant changes are picked up automatically by %autoreload 2.)

# 3. Run the smoke test to verify nothing structural broke:
pytest tests/ -q

# 4. Run the relevant notebook for a live LLM check:
#    notebooks/test_chat_mode.ipynb  —  or  —  notebooks/test_report_mode.ipynb
Section 6

Gotchas

  • Placeholder substitution. $question, $findings, $domain in specialist: templates are replaced via simple .replace() in report_loader.py. Don't use ${question} — no shell expansion, no Python format strings.
  • YAML cache. _load_prompts() caches the file read. Call report_loader.reload() after editing prompts.yaml if the process is long-running.
  • synthesis_report can be string or dict. The dict form supports per-section prompts (early_risk_prediction_section: { report_instructions: ... }) plus a final formatter_report. See config/pillars/credit_risk.yaml for the dict pattern.
  • Pillar overlay edits only affect that pillar. If a change should apply everywhere, it belongs in prompts.yaml (common.* or {mode}.*), not in a pillar file.
  • Skill system_prompt is truncated to 150 chars when shown to the orchestrator's plan_team() — see _build_specialist_descriptions in orchestrator/orchestrator.py. Keep the most distinctive part in the first sentence.
  • Case-ID tokens in prompt text get scrubbed to <case> before the LLM sees them (firewall layer). Don't hard-code a real case ID in any prompt — it'll show as <case> at runtime.
  • Sub-questions. When plan_team() decomposes the user question, each specialist receives a tailored sub-question via agent.run(sub_question, mode, root_question=original). The base agent's prompts include both the root question and the sub-question as context — see agents/base_agent.py for the exact framing.
