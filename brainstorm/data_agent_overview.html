The case ID lives on the gateway — implicit in every query; when it surfaces to the LLM, the format keeps it firewall-safe

Three places the case ID is handled
1
Set once at session start
main.py · before any LLM call
The CLI picks the case, then gateway.set_case(case_id) stores it on the module-singleton gateway. Every subsequent read is implicitly scoped.
gateway.set_case("<case_id>") # logger.log("case_selected", ...) init_tools(gateway, catalog)
2
Not a tool parameter
tools/data_tools.py
Tool signatures only accept table_name, optional filter, and optional columns. The specialist prompt never sees a case_id argument — can't be leaked, can't be wrong.
def query_table( table_name: str, filter_column: str = "", filter_value: str = "", columns: str = "", ) -> str:
3
Stripped from rows
data/gateway.py · from_generated
During the pivot, case_id is explicitly removed from each row — it is implicit from the case context. The LLM sees data columns only, not identifiers.
row = {c: cols[c][i] for c in col_names if c != "case_id"}

Would trigger Raw customer ID
A bare customer or account number (11–16 digits) would be masked by SafeChain on every call, blocking correct data joins. The design avoids carrying such identifiers into prompts.

Summary. The data agent keeps the case ID mostly out of band: stateful on the gateway, absent from tool parameters, stripped from data rows. It surfaces only in the list_available_tables() response header, and the CASE-{seq:05d} format keeps that surface firewall-safe (5 digits, below both thresholds). Long digit runs in any other field (payment amount, card number, account reference) are masked by SafeChain on every call — so even if data does carry sensitive-looking digits, the LLM never sees the raw value.
