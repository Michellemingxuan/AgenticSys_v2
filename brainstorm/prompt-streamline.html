How Specialist & Synthesis Prompts Are Assembled · Report & Chat Modes

Prompt Source Files
config/prompts/common.yaml
Shared format rules for all specialist outputs in both modes. Controls how evidence is cited, how thresholds are referenced, how absent data is flagged. Edit this to change the "voice" of all specialists at once.
common_format_instructions
config/pillars/{pillar}.yaml
Per-specialist instructions that vary by pillar. Each domain (bureau, crossbu, modeling, ...) has a report_instructions field telling the specialist what to compare, which thresholds to flag, and what cross-references to make — specific to this pillar's analytical lens.
bureau.report_instructions crossbu.report_instructions modeling.report_instructions ...
config/prompts/report.yaml
Report mode prompts. Specialist section structure (Key Findings / Evidence / Implications as bullets). Synthesis report assembly (Detailed Findings / Cross-Domain / Items Requiring Attention / Summary).
specialist synthesis
config/prompts/chat.yaml
Chat mode prompts. Specialist answer format (concise, direct, 3-8 sentences). Synthesis assembly (1-2 paragraphs, lead with answer, cite evidence, flag attention items).
specialist synthesis
Specialist Prompt Assembly

When a specialist produces its output, the prompt is assembled from 3 layers. The same Base Agent receives different prompts depending on mode and pillar — no code changes needed.

1
Common Format Rules
config/prompts/common.yaml
  • Bullet format: one idea per bullet, self-contained
  • Use exact variable/metric names as provided
  • Provide short description + original name in parentheses
  • Cautious language: "suggests...", "indicates..."
  • Include values, ranges, dates wherever available
  • Bold minimal meaningful phrases only
  • No tables unless explicitly required
+
2
Mode Instructions
config/prompts/{report|chat}.yaml → specialist
  • Report: ## Domain / Key Findings / Evidence / Implications
  • Chat: Direct answer, 3-8 sentences, concise
  • JSON output schema for structured parsing
  • Placeholders: $question, $findings, $domain
+
3
Pillar-Specific Instructions
config/pillars/{pillar}.yaml → {domain}.report_instructions
  • What scores/metrics to compare against thresholds
  • What cross-references to make with other domains
  • What specific patterns to flag
  • Different per pillar × domain combination
  • Optional — omitted if not set
Example: Bureau specialist under credit_risk pillar in report mode receives:

Layer 1: "Use bullet points, one idea per bullet, cite exact variable names, use cautious language..."
Layer 2: "Produce a structured report section: ## Bureau / Key Findings / Evidence / Implications..."
Layer 3: "Compare FICO, SBFE, LN, CSS, FSS, Paydex scores against thresholds. Flag if Amex is primary lender. Note judgment and lien counts..."
Report Mode vs Chat Mode — Side by Side
📄 Report Mode
Full structured report for case review — detail for analysts, summary for supervisors.
Specialist Output
Structured section per domain:
## Domain Name
Key Findings: 3-5 bullets with data citations
Supporting Evidence: Exact sources, trends, recency
Implications: Risk meaning, suggested actions
Synthesis Output
## Detailed Findings — per-domain subsections with bullets, cross-validated notes
## Cross-Domain Insights — patterns across specialists
## Items Requiring Attention — open conflicts + absence signals
## Summary — 3-5 sentences for supervisors: posture, concerns, next steps
Contradiction Handling
Resolutions woven into domain findings with "Cross-validated with [other domain]" notes. No separate contradictions section.
💬 Chat Mode
Concise direct answer for specific reviewer questions.
Specialist Output
Direct answer in 3-8 sentences:
Lead with the answer, cite data values, note caveats and gaps.
Synthesis Output
1-2 paragraphs:
Lead with direct answer, weave in evidence from specialists, note contradictions, flag attention items.
Contradiction Handling
If resolved: state the resolution inline. If unresolved: note it as a caveat.
Pillar-Specific Instructions (credit_risk example)

Each specialist gets pillar-specific instructions on top of the common + mode prompts. These tell the specialist what to look for under this pillar's analytical lens. The same Bureau specialist gets different instructions under credit_risk vs. escalation vs. cbo.

Bureau
  • Compare FICO, SBFE, LN Credit, LN Blended, CSS, FSS, Paydex scores against their thresholds
  • Highlight any score that has declined over time
  • Flag if Amex is the primary lender (≥ 40% exposure)
  • Note judgment and lien counts — even 1 is significant
  • If tradeline-level data shows 90D+ DPD, call it out explicitly
Cross-BU
  • List each card with type, limit, balance, and status
  • Flag any card with account_status other than "current"
  • Calculate total internal exposure across all cards
  • Note merchant relationship with Amex (revenue offset)
  • Flag if past_delinquencies_12m > 0 on any card
Modeling
  • Reference score drivers — which variables push risk up vs down
  • Compare credit_loss_prob and tot_struct_risk_score against thresholds (10 and 20)
  • Note positive_events count — low count may indicate stagnating relationship
  • Flag any variable where the value crosses its risk threshold
Spend & Payments
  • Summarize spend volume and merchant concentration
  • Flag transactions with merchant_risk_score > 0.7
  • Note if spend_concentration > 2.4 or rnn_spend_score > 0.017
  • Flag if spend_divergence_index < 1.0 (abnormal pattern)
  • Look for spend spikes or sudden changes in merchant mix
WCC
  • List each active flag with type, date, and severity
  • Critical and high severity require explicit callout
  • Note if multiple flags cluster in time (acute event)
  • Cross-reference flags with other domain findings
Capacity & Affordability
  • State DTI ratio and compare against 0.43 threshold
  • Compare total_debt to income_est
  • Flag if total debt exceeds annual income
  • Note whether income estimate appears current or stale
