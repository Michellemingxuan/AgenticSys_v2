Shared Skills
Six skills across all specialists. Data Request and Synthesize are foundational (data in → findings out). Report and Answer produce output. Question and Reflection drive the peer review loop.
Foundational
Data Request
Describes what data is needed using semantic variables (e.g. "delinquency count", not exact column names). May optionally hint at tables. The Text-to-SQL skill resolves the actual table and column mapping. If a data unavailable notice is returned, the specialist flags what information is missing.
Synthesize
Ingests structured data returned by Data Agents and synthesizes it into analytical findings — identifies patterns, anomalies, trends, and gaps. Can be re-triggered during Reflection when additional data is fetched.
Output
Report 📄 Report Mode
Produces a standardized section narrative from synthesized findings. Enforces consistent structure: Key Findings → Supporting Evidence → Risk Implication. Designed for assembly into the final case report.
Answer 💬 Chat Mode Peer Review
Answers the sub-question distributed by the orchestrator. Also used during peer review to respond to questions raised by reviewers — delivers evidence-backed replies.
Peer Review
Question
Reads a peer specialist's output and raises questions — flags contradictions, challenges unsupported claims, identifies missing evidence, or highlights conflicting risk signals. Waits for the peer's response and determines whether the issue is resolved or requires further challenge.
Reflection
Triggered when questions are raised by a peer reviewer. Stays focused on the original question from the orchestrator. Addresses peer challenges by sending additional Data Requests for missing evidence, revising Synthesize findings, then triggering Answer to send a grounded response back to the reviewer.
⇄   Pairwise Review — Iterative Review Loop (Both Modes)
1
First-Pass Generation
All called specialists run Data Request → Synthesize → Report (report mode) or Data Request → Synthesize → Answer (chat mode) in parallel. Each produces an independent output with no knowledge of peers.
Bureau ✓ CrossBU ✓ Modeling ✓ … ✓
2
Pair Assignment
Orchestrator assigns review pairs. Each pair shares context for the review loop.
Bureau ⇄ CrossBU
Modeling ⇄ Cap. & Afford.
Spend & Pmts ⇄ Cust. Rel.
WCC → Standalone
3
Review Loop — Runs Until Clear
Repeats
❓
Question skill — reviewer reads peer's output, raises challenges, flags contradictions or missing evidence, waits for response
↓
🔄
Reflection skill — the challenged peer stays focused on the original question, sends additional Data Requests if needed, revises Synthesize findings, triggers Answer to respond
↓
✓
Question skill evaluates — reviewer determines whether the response resolves the issue. If not, raises follow-up questions. If clear, passes.
Context management: Each pair maintains a shared context window across rounds — original outputs, all questions raised, all reflections and responses. The orchestrator manages context to prevent drift and ensure later rounds reference earlier resolutions.
4
Final Refined Output
Each specialist produces its final revised output incorporating all peer feedback. Refined outputs are returned to the orchestrator's Synthesize skill for final assembly.

Layer 3 — Data Infrastructure (Shared Across All Pillars)
Repository Agent — Gatekeeper
Request processing pipeline (every Data Request passes through in order):
1
Pillar Access Control — checks silenced tables/schemas for the requesting pillar; rejects or filters fields restricted at the pillar level
2
Access Gating — validates request against user role permissions
3
Redaction — strips PII and sensitive fields before forwarding to data agents
Administrative Skills
Catalog Governance
Register, update, or retire tables and schemas in the shared Data Catalog. Maintains descriptions, schema metadata, and data lineage.
Pillar Access Control
Silence specific tables or schemas per pillar based on compliance rules. Configured via pillar YAML silenced_tables / silenced_schemas — restricts what a pillar can request without removing data from the catalog.

Shared Skills — Data Agents
All data agents expose the same four skills.
Text-to-SQL
Reads the Data Catalog's schema descriptions to intelligently match the specialist's semantic variables to real tables and columns. Resolves table selection when the specialist doesn't specify one. If no match found, returns a structured data unavailable notice.
Execute
Run generated code in a sandboxed environment. Retry on recoverable failures.
Retrieve
Return structured results to the requesting specialist via the Repository Agent.
Analyze
Optional: produce a brief statistical summary or anomaly flag on retrieved data before returning.
